# Vibe Coding 指导手册

> "Vibe Coding" 是一种由 Andrej Karpathy 提出的编程理念：让 AI 协助完成 80% 的代码工作，开发者专注于 20% 的核心逻辑和创意。

---

## 1. Vibe Coding 核心原则

### 1.1 不是"AI 写代码你复制"

```
❌ 错误方式:
让 AI 生成完整代码 → 复制粘贴 → 不理解原理 → 无法调试

✅ 正确方式:
让 AI 生成 → 理解每行逻辑 → 修改适配 → 调试成功 → 掌握
```

### 1.2 人机协作模式

```
┌─────────────────────────────────────────────────────────────┐
│                        你的角色                              │
│  - 需求定义 (告诉 AI 要做什么)                               │
│  - 架构决策 (选择技术方案)                                   │
│  - 边界判断 (决定什么能做/什么不能做)                        │
│  - 质量把控 (审查 AI 输出的代码)                             │
│  - 调试修复 (处理边界情况和错误)                             │
└─────────────────────────────────────────────────────────────┘
                              ▲
                              │ 指令/反馈
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        AI 的角色                             │
│  - 样板代码生成 (减少重复劳动)                               │
│  - 模式匹配 (快速实现常见模式)                               │
│  - 文档生成 (解释代码意图)                                   │
│  - 测试建议 (补充边界用例)                                   │
│  - 重构优化 (改善代码结构)                                   │
└─────────────────────────────────────────────────────────────┘
```

## 2. Mems 项目的 Vibe Coding 流程

### 2.1 阶段一：需求到原型

**目标**: 快速验证想法

**你的任务**:
1. 写出 3-5 句需求描述
2. 选择技术栈组合
3. 确认核心数据流

**AI 协助**:
- 生成项目脚手架
- 创建基础文件结构
- 实现最简单的 Hello World

**Mems 示例**:
```
你: "我需要一个四层记忆系统，用 SQLite 存关系数据，用 Qdrant 存向量，
    支持记忆摄取和检索，帮我搭建基础结构"

AI 生成:
├── pyproject.toml
├── docker-compose.yml
├── src/mems/
│   ├── main.py
│   ├── config.py
│   ├── models.py
│   └── database.py
└── requirements.txt (基础版)
```

### 2.2 阶段二：增量实现

**目标**: 功能逐步叠加

**你的任务**:
1. 每次只实现一个功能点
2. 写清楚输入和输出
3. 验证上一个功能正常工作

**AI 协助**:
- 实现单个 API 路由
- 编写 Service 层逻辑
- 生成测试用例

**Mems 示例**:
```
第一轮:
你: "实现 /ingest 接口，把内容同时写入 SQLite 和 JSONL"
AI: 生成 ingest.py + 修改 main.py

第二轮:
你: "实现向量存储，用 HTTP 调用 Qdrant REST API"
AI: 生成 vector_service.py (REST API 版)

第三轮:
你: "实现 /search 接口，调用 embedding 服务和向量检索"
AI: 修改 search.py
```

### 2.3 阶段三：调试优化

**目标**: 修复问题，提升质量

**你的任务**:
1. 提供具体错误信息
2. 描述期望行为
3. 逐步缩小问题范围

**AI 协助**:
- 分析错误原因
- 提供修复建议
- 解释不工作的原理

**Mems 调试示例**:
```
你: "Qdrant 查询返回 404，但 collection 存在"
AI: "这是因为 qdrant-client 1.17.1 的 query_points API 
     在某些版本有 bug。解决方法：改用 REST API 直接调用..."

你: "向量维度不匹配"
AI: "BAAI/bge-small-zh-v1.5 输出 512 维，不是 384。
     需要在 create_collection 时动态获取维度..."
```

## 3. 有效提问技巧

### 3.1 好的提问模板

```markdown
## 背景
(简短描述你在做什么项目)

## 需求
(你想要什么)

## 已尝试
(你已做了什么)

## 问题
(具体什么错误/不符合预期)

## 环境
(关键配置: Python 版本, 操作系统, 相关包版本)
```

### 3.2 Mems 项目示例

**❌ 差劲的提问**:
```
帮我实现记忆系统
```

**✅ 好的提问**:
```
## 背景
我正在开发 Mems 分层记忆系统，使用 FastAPI + SQLModel

## 需求
实现 /search 接口，需要：
1. 先用 embedding 服务把 query 转为向量
2. 调用 Qdrant 检索相似内容
3. 同时查询 L2 语义知识
4. 合并结果按 score 排序

## 已尝试
写了一个版本，但返回空结果

## 问题
Qdrant 返回的 payload 是 null，但我知道数据是写入成功的

## 环境
- Python 3.12
- qdrant-client 1.17.1
- Qdrant 1.7.4 (Docker)
```

## 4. 代码审查清单

AI 生成的代码，需要你确认以下几点：

### 4.1 功能正确性
- [ ] 输入输出符合预期
- [ ] 错误处理合理
- [ ] 边界情况有考虑

### 4.2 安全
- [ ] 无硬编码密钥
- [ ] 输入校验
- [ ] 无 SQL 注入风险

### 4.3 性能
- [ ] 异步/同步选择正确
- [ ] 无明显性能问题
- [ ] 资源正确释放

### 4.4 可维护性
- [ ] 命名清晰
- [ ] 注释适当
- [ ] 符合项目规范

## 5. Mems 项目调试案例

### 案例 1: Qdrant REST vs SDK

**问题**: qdrant-client 的 `query_points` 方法一直返回 404

**调试过程**:
```
1. 用 curl 直接测试 REST API → 成功
2. 确认 collection 存在 → 是
3. 确认向量维度正确 → 512 维
4. 确认 SDK 版本 → 1.17.1 (有已知 bug)
5. 决定: 绕过 SDK，直接用 httpx 调用 REST API
```

**学到的**: 不要盲目信任 SDK，有问题先用 REST API 验证

### 案例 2: 向量维度不匹配

**问题**: 创建 collection 用 384 维，但 bge-small-zh 输出 512 维

**调试过程**:
```
1. 收到 "expected dim: 384, got 512" 错误
2. 检查 embedding 模型文档 → BAAI/bge-small-zh 是 512 维
3. 修改代码：upsert 时动态获取维度
```

**学到的**: 不同 embedding 模型维度不同，不能硬编码

## 6. Vibe Coding 推荐工作流

### 6.1 每日流程

```
早上:
1. 明确今天的目标 (1-2 个功能点)
2. 先运行项目确保基础正常
3. 列出具体需求清单

下午:
4. 实现一个功能 → 测试
5. 实现第二个功能 → 测试
6. 代码审查 → 提交

晚上:
7. 记录遇到的问题和解决思路
8. 规划明天的工作
```

### 6.2 迭代速度

| 阶段 | 预期时间 | 验证方式 |
|------|----------|----------|
| 新功能实现 | 30-60 min | API 测试 |
| Bug 修复 | 15-30 min | 复现问题 |
| 重构优化 | 60+ min | 全部测试 |

## 7. 常见陷阱

### 7.1 过度依赖 AI

```
❌ 症状: 每次写代码都要问 AI，连简单 if/else 都不会
✅ 解法: 掌握基础，AI 用于复杂模式和重复劳动

❌ 症状: 不理解 AI 生成的代码就复制使用
✅ 解法: 至少读一遍，标记不理解的地方问 AI 解释

❌ 症状: 完全信任 AI，不做代码审查
✅ 解方: 用第 4 节的清单逐项检查
```

### 7.2 需求不清晰

```
❌ 症状: "帮我实现搜索功能" → AI 猜错了
✅ 解法: 具体说明输入输出和边界

❌ 症状: 同时提 10 个需求 → 哪个都做不好
✅ 解法: 一次只提 1-2 个，验证后再继续
```

### 7.3 不验证就推进

```
❌ 症状: 实现 5 个功能后测试，发现第一个就有问题
✅ 解法: 每实现一个功能，立即测试验证
```

## 9. 进阶: 成为更好的 Vibe Coder

### 9.1 基础知识要求

即使 AI 能写代码，你仍需掌握:
- [ ] Python 基础 (语法、数据结构、异步)
- [ ] HTTP 协议 (REST API)
- [ ] 数据库概念 (SQL、索引)
- [ ] 向量/Embedding 基础概念

### 9.2 经验积累

每完成一个功能:
1. 记录遇到的问题
2. 总结解决方法
3. 提炼可复用的模式

### 9.3 持续学习

- 关注 AI 工具更新
- 学习新框架/新范式
- 理解 AI 的局限性

---

## 10. 总结

Vibe Coding 不是让 AI 替代你编程，而是:

```
AI (80% 样板代码) + 你 (20% 核心决策) = 更好的生产力
```

**核心公式**:
```
好结果 = 清晰需求 + 及时验证 + 主动理解 + 持续改进
```
